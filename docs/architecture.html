System Overview

The application has three main operating loops. First, dbt builds a local semantic warehouse and metadata artifacts. Second, metadata is embedded into Qdrant so the agent can retrieve official table, column, semantic-model, and metric context. Third, Streamlit routes users into Ask, Library, Dashboard, Experiments, and About workflows.

  • Experiments are observational, not randomized A/B tests.
  • Approved reports and snapshots persist locally under agent_store/.

Ask Trust Loop

The Ask tab calls stream_agentic_events. The orchestrator handles follow-up condensation, intent routing, semantic grounding, memory, fast paths, the LangGraph analysis loop, and snapshot checks.

1. Question Streamlit Ask

User asks in plain English. Prior turns can be passed as history.

2. Condense Follow-up resolver

Ambiguous follow-ups become standalone questions before routing.

3. Route Intent classifier

Definition, quick metric, dashboard request, or deeper analysis.

4. Ground Semantic layer lookup

Metric definitions are selected from dbt's official semantic manifest.

5. Recall Report memory

Approved prior reports are recalled from SQLite or an optional vector hook.

6. Analyze LangGraph loop

Plan, retrieve, generate SQL, validate, execute, reflect, and synthesize.

7. Verify Snapshots

First numeric finding can be compared against trusted historical baselines.

8. Save Library

Useful reports and approved generated dashboards are persisted for reuse.

LangGraph Analysis Engine agent

  • plan_analysis creates hypotheses and ordered sub-questions.
  • run_query retrieves context, generates SQL, validates it, dry-runs when useful, executes, and records findings.
  • reflect can enqueue bounded follow-up questions.
  • synthesize_report writes a cited final answer from returned rows.

Bounded Runtime limit

  • MAX_SUB_QUERIES defaults to 8.
  • Each SQL sub-question gets one draft plus one repair attempt.
  • Detail queries receive a default limit when not aggregate queries.

Fast Paths routing

  • Metric definitions return semantic-layer contracts directly.
  • Simple AI-user questions use trusted dashboard KPI helpers.
  • Dashboard requests produce declarative dashboard specs for preview and approval.

LangGraph And Query Routing

There are two layers to understand. The outer agentic router decides whether a question needs the full analysis graph at all. The inner LangGraph only runs for open-ended analysis questions that need planned, validated SQL over the warehouse.

Outer router: every Ask question starts here

Optional chat history can come with it.

Example question Intent Response mode What runs
"What is daily active users?" metric_definition_question definition Semantic-layer lookup returns the official metric contract. The SQL graph does not run.
"How many users use AI?" simple_metric_question quick_answer Trusted dashboard KPI helpers answer directly with AI users, total users, and adoption rate.
"Monitor daily active users by week" dashboard_request dashboard_spec A declarative dashboard spec is drafted for preview, optional JSON editing, and approval.
"Why did AI credit consumption change?" analysis standard_analysis The full LangGraph plan-query-reflect-synthesize loop runs with guarded SQL.
"Give me a detailed deep dive on AI usage" analysis deep_dive The same analysis graph runs, but the final response is expected to be more complete.

Inner LangGraph: only for analysis questions

Node 1 plan_analysis

LLM creates hypotheses and ordered sub-questions.

->
Node 2 run_query

Retrieve Qdrant context, draft SQL, validate, execute, record finding.

->
Guardrail validate_sql

Only read-only, allow-listed dbt model references pass.

->
Node 3 reflect

Checks whether evidence is enough or bounded follow-ups are needed.

->
Loop More run_query?

Runs while work remains and max_sub_queries is not reached.

->
Node 4 synthesize_report

Writes the final answer with cited findings and uncertainty.

Conditional Edges LangGraph

  • plan_analysis goes to run_query if planned work exists, otherwise reflect.
  • run_query repeats until all planned sub-questions have run or the query budget is reached.
  • reflect may append follow-up sub-questions, then route back to run_query.
  • When no more work is pending, synthesize_report runs and the graph ends.

Query Safety Path SQL

  • Each sub-question retrieves dbt context from Qdrant before SQL generation.
  • Generated SQL is stripped from fences and given a default limit when needed.
  • Validation blocks mutations and tables outside the manifest-derived allow-list.
  • BigQuery can dry-run for bytes; DuckDB opens the local database read-only.

Events The UI Sees stream

  • intent, response_mode, and semantic_layer appear before any analysis work.
  • Analysis emits plan, one or more query events, reflect, and synthesize.
  • Open-ended analysis adds a snapshot_check before final_response.

Feature Surfaces

The Streamlit UI is not only chat. It exposes several working surfaces, each with a different trust model and persistence path.

Ask

Conversational agent with streamed reasoning events, collapsible SQL and sources, model usage, report save action, and generated dashboard preview.

Library

Saved reports and approved dashboards grouped by theme, month, or type. Reports live in SQLite; dashboards live as JSON specs.

Dashboard

Activation & Conversion surface over AHA models, with filters, KPI tiles, lift leaderboard, activation depth, churn watch, retention, and funnel views.

Generated Dashboards

Dashboard specs are declarative. Chart specs map to trusted helpers such as daily activity, feature usage, token usage, funnels, and retention.

Experiments

Observational cohort comparisons with continuous tests, proportion tests, confidence intervals, effect sizes, and an LLM-drafted recommendation.

Model Routing

Chat model creation is provider-agnostic. By default, the app uses OpenAI-compatible chat via gpt-4.1-mini, while embeddings remain OpenAI text-embedding-3-small so the existing Qdrant collection stays compatible.

Task Default Behavior Relevant Files
Condense Resolve chat follow-ups into standalone questions. analytics_agent/ask/graph.py, model_profiles.yml
Plan, SQL, Repair, Reflect, Synthesize Routed per task when MODEL_ROUTING_ENABLED is enabled; otherwise uses the global provider and model. analytics_agent/ask/model_router.py, analytics_agent/core/llm.py
Experiment Narrative LLM summarizes statistical output with observational caveats and a ship/no-ship/inconclusive recommendation. analytics_agent/experiments/
Dashboard Chart Repair LLM can repair an unsupported chart spec, but rendering still maps to trusted helper functions. analytics_agent/dashboard/render.py
Cost Tracking Token usage is read from provider metadata when available, otherwise estimated at roughly four characters per token. analytics_agent/ask/model_router.py

Guardrails

The app is designed so generated content passes through narrow, testable boundaries before it touches the warehouse or saved artifacts.

SQL Validation read-only

  • Only SELECT and WITH are allowed.
  • Mutation keywords such as insert, update, delete, drop, merge, truncate, alter, and create are blocked.
  • Queries must reference at least one allowed dbt model.
  • Tables outside the manifest-derived allow-list are rejected.
  • Backend-specific fully qualified identifiers are required.

Backend Safety runner

  • DuckDBRunner opens the database read-only per query.
  • BigQueryRunner supports dry-run byte estimates for optional cloud execution.
  • Dashboard and experiment code use the same runner abstraction.

Generated Dashboard Safety mapping

  • Specs are JSON-like declarations, not executable SQL.
  • Chart types are limited to line, bar, funnel, and heatmap.
  • Rendering maps metrics and dimensions to author-written helper queries.

Experiment Caveats stats

  • No assignment column means no true randomized A/B testing.
  • Cohorts are observational and may be confounded by self-selection.
  • Continuous outcomes use Welch t-test, Mann-Whitney U, Cohen's d, and CI.
  • Rates use two-proportion z-test, chi-square, risk difference, and CI.

Snapshot Checks drift

  • Trusted KPI baselines can be seeded from dashboard helpers.
  • Current numeric findings can be compared to latest snapshots.
  • Default drift threshold is 15 percent.

Human Approval library

  • Reports and dashboards can be reviewed before reuse.
  • Generated dashboard JSON can be edited before approval.
  • Approved artifacts become Library items.

Evals And Tests

Evals check the high-level agentic trust loop, while unit tests cover the implementation details that make the loop dependable.

Eval Scorecard routing

evals/run_evals.py loads evals/cases.yaml, runs deterministic intent classification, response-mode selection, semantic metric lookup, and source-table checks.

  • Definition case: daily active users.
  • Quick metric case: AI users.
  • Dashboard case: monitor daily active users weekly.
  • Analysis case: AI credit consumption changes.

Unit Test Coverage unittest

  • Agent flow and orchestrator behavior.
  • SQL validation and backend runner behavior.
  • Semantic layer contracts and model router selection.
  • Dashboard helpers, dashboard rendering, snapshots, and memory.
  • Experiment statistics and UI helper functions.
Recommended verification commands: python -m unittest discover -s tests, python evals/run_evals.py, and python embedding_script.py --local-smoke-test.

Storage And Configuration

Environment config

  • BACKEND=duckdb by default, bigquery optional.
  • OPENAI_API_KEY is required for default chat and embeddings.
  • QDRANT_URL, QDRANT_API_KEY, and QDRANT_COLLECTION_NAME configure retrieval.
  • LLM_PROVIDER can be openai, claude, kimi, or openrouter.

Local Files artifacts

  • zerve_transform/zerve.duckdb stores materialized dbt tables.
  • zerve_transform/target/manifest.json supplies model allow-listing.
  • zerve_transform/target/semantic_manifest.json supplies official metric contracts.

Agent Store persistence

  • agent_store/reports.db stores saved report records.
  • agent_store/snapshots.db stores trusted metric baselines.
  • agent_store/dashboards/*.json stores approved dashboard specs.

Change Map

Use this section when deciding where to make future changes. It points to the module boundary most likely to own each kind of work.

Changing Data Definitions

  • New marts or columns: edit zerve_transform/models/** and matching schema.yml.
  • New official metrics: update semantic models and metrics in marts schema YAML, then rebuild semantic manifests.
  • New AHA/dashboard source fields: update dbt models and dashboard helper queries together.

Changing Agent Behavior

  • Intent or response mode: analytics_agent/ask/graph.py and evals/cases.yaml.
  • Model choices or costs: model_profiles.yml and analytics_agent/ask/model_router.py.
  • Retrieval content: embedding_script.py and dbt metadata.

Changing Safety Rules

  • SQL constraints: analytics_agent/core/sql.py and tests/test_sql_validation.py.
  • Backend behavior: analytics_agent/core/runners.py, bigquery_client.py, and backend tests.
  • Snapshot drift logic: analytics_agent/ask/snapshots.py.

Changing Product Surfaces

  • Ask, Library, Dashboard, Experiments, About: one module per tab under analytics_agent/ui/ (wired by streamlit_app.py).
  • Dashboard data helpers: analytics_agent/dashboard/queries.py.
  • Generated dashboard rendering and persistence: analytics_agent/dashboard/render.py.
  • Experiment statistics and narrative: analytics_agent/experiments/.
